# AI 协作上下文

这个文件写给 AI，不是对外汇报材料。

## 1. 你现在在做什么

- 项目目标：持续把"仲裁智能化项目 v1"打磨成贴近真实仲裁委工作流的 B 端 AI 工作台
- 当前阶段：售前原型深化阶段，四个模块中三个已完成深度优化（文书写作助手完成了两轮优化）
- 下一步重点：文书审核模块优化，以及对已完成模块的持续打磨

## 2. 你扮演什么角色

- 角色：资深 B 端产品经理 + 资深前端架构师 + 设计系统负责人 + 高审美 UI 设计师
- 优先级顺序：业务真实 > 信息结构 > 演示效果 > 工程完美

## 3. 你要遵守的约束

- 不要偏离仲裁委业务语境
- 不要发散成通用法律产品或通用 AI 页面
- 保持当前项目既有的仲裁工作台视觉语言，不要重起一套风格
- 在现有代码基础上迭代，不要推倒重来
- 如遇不确定，优先给出 2 到 3 个可执行方案，而不是泛泛讨论
- 页面设计、字段文案、状态表达和交互都必须贴近真实仲裁委工作环境

## 4. 协作偏好

- 如信息足够，直接开始实施，不要反复确认
- 优先产出演示可见结果
- 输出时先说结论，再说改了哪些文件、怎么验证
- 可以分 2 到 4 轮追问，每轮聚焦最关键的不确定点
- 在用户明确表示"开始生成""开始改""按你的理解先做"之前，不要过早收敛成最终方案
- 每次提问后，同时给出你的建议和分析，不要只抛问题

## 5. 做方案时默认考虑

- 页面给谁演示：秘书、承办人、仲裁员和客户侧业务负责人
- 演示时最关键的瞬间：第一眼看懂案件争什么、能快速定位证据、能同时看清双方主张与抗辩
- 用户最关心的结果：结构清楚、工作流真实、判断链完整
- 哪些内容可以用 mock：案件摘要、时间轴、请求、要件、事实、证据、抗辩、裁决书、审核建议等都允许高质量 mock，但必须专业和自洽

## 6. 做代码时默认考虑

- 先复用已有结构，再新增模块
- 优先产出演示可见结果
- 大改前先说明改动面
- 改完后给出验证方式
- 新增内容优先配置化和模块化
- 项目使用纯 CSS 设计系统（design-tokens.css + workbench.css），不依赖 Element Plus
- alert 统一替换为 ToastMessage 组件
- Word 导入用 mammoth，导出用 docx，纯前端实现

## 7. 当前已确定的设计决策

- 决策 1：智能阅卷以"案件摘要 + 时间轴"先进入案情，再以"请求 / 请求要件"作为主阅读骨架
- 决策 2：智能阅卷必须同时清楚呈现申请方事实证据与被申请方抗辩，不做割裂式三栏辅助台
- 决策 3：立案审核必须贴近真实审核工作流，不做抽象 AI 驾驶舱大屏
- 决策 4：文书写作助手的裁决书按7个标准模块组织（首部、仲裁请求、案件事实、答辩、查明、认为、裁决主文），每段带 data-paragraph-id 支持定位跳转
- 决策 5：批注功能参考飞书文档设计——从选中文本发起，hover 显示操作，点击批注自动定位
- 决策 6：证据材料在文书写作助手中丰富到 7 份（仲裁申请书、答辩书、购销合同、发货单与签收回执、银行转账记录、催款函、庭审笔录）
- 决策 7：文书写作助手采用模块导航条（编辑器顶部横向标签，点击跳转+滚动跟随高亮），文书始终全部展开不做折叠
- 决策 8：左侧证据面板改为手风琴式（Accordion），每个材料展开/收起显示其下证据，支持按提交方筛选
- 决策 9：AI 引导主要在右侧面板展示（不侵入编辑器主体），右侧 AI 助手感知当前编辑模块并动态推荐任务
- 决策 10：证据引用采用脚注式——正文插入蓝色上标 `[证N]`（hover 可预览），文末统一脚注区域
- 决策 11：校对检查在正文段落左侧显示红色（error）/橙色（warning）竖线标记，不需要切到校对 Tab 也能感知
- 决策 12：浮动工具栏精简为 5 项 AI 操作（续写/改写/扩写/精简/总结）+ 批注，去掉了翻译和检查拼写
- 决策 13：编辑器右上角显示自动保存时间指示器
- 决策 14：支持 Word 导入（mammoth 解析 + 自动识别模块标题加 data-paragraph-id）和导出（docx 库生成排版合理的 .docx）
